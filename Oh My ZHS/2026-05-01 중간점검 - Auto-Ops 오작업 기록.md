# Auto-Ops 중간점검: Practical Tools Studio 구축 기록

## 한 줄 요약

`auto-ops.lotiony.com`을 블로그 중심 사이트가 아니라 "실제로 쓸 수 있는 practical tools studio"로 전환하는 작업을 시작했고, 현재까지 총 17개 도구의 registry/page shell을 만들고 그중 17개 모두를 브라우저에서 동작하는 인터랙티브 도구로 구현했다.

현재 브랜치: `feat/agentic-auto-ops-plan`

마지막 검증:

```bash
pnpm lint
pnpm build
```

결과: 성공. Next.js build에서 47개 static page 생성 확인.

---

## 왜 이 작업을 시작했나

초기 방향은 기존 블로그 표면 위에 콘텐츠를 쌓는 방식에 가까웠다. 하지만 `auto-ops.lotiony.com`의 목표는 단순 블로그나 AI wrapper가 아니라, AI 에이전트가 빠르게 만들고 검증해 나가는 practical tools studio에 더 가깝다.

그래서 60일짜리 긴 waterfall 계획은 폐기하고, 7~10일 안에 MVP/AdSense-ready에 가까운 표면을 만드는 압축 실행 방식으로 바꿨다.

핵심 방향은 다음과 같다.

1. 사이트 첫인상은 "글 모음"이 아니라 "쓸 수 있는 도구 모음"이어야 한다.
2. 각 도구 페이지는 thin content가 아니어야 한다.
3. 각 도구에는 실제 입력 UI, 예시, 설명, FAQ, 관련 링크가 있어야 한다.
4. Claude Design 결과를 기다리지 않고, IA/data model/tool logic부터 먼저 밀어붙인다.
5. 작은 단위로 구현하고, lint/build 검증 후 커밋한다.

실제 오류:

```text
The 'gpt-5.2-codex' model is not supported when using Codex with a ChatGPT account.
```

그래서 하네스 문서에 이 사실을 반영했고, 반복 재시도 대신 controller 세션에서 직접 구현하고 검증하는 fallback을 사용했다.

블로그 글감 관점에서는 이 지점이 중요하다. "AI 에이전트 기반 개발"은 멋진 말이지만, 실제로는 모델 권한, 계정 타입, 세션 설정, fallback 전략 같은 운영 문제가 바로 나온다. 이번 작업은 그 문제를 우회하면서도 산출물을 계속 쌓는 쪽으로 진행했다.

## 구현된 사이트 구조

### 핵심 라우트

현재 구현/사용 중인 주요 라우트:

- `/`
- `/tools`
- `/tools/[slug]`
- 기존 posts/tags/feed/sitemap/robots 라우트

`/tools/[slug]`는 registry-driven dynamic route로 구성했다. `src/lib/tools.ts`의 tool registry에서 slug, title, description, SEO metadata, examples, FAQ, related tools 정보를 가져오고, route에서는 slug에 맞는 interactive widget을 꽂는 방식이다.

이 구조의 장점:

- 도구가 늘어나도 페이지 생성 구조가 단순하다.
- SEO metadata와 콘텐츠 구조를 registry에서 일관되게 관리할 수 있다.
- 모든 도구를 SSG로 생성할 수 있다.
- 나중에 guide/content registry와 internal linking을 붙이기 쉽다.

### Tier 2: Developer / automation utilities

#### 7. JSON/YAML Validator

파일:

- `src/components/tools/JsonYamlValidator.tsx`

기능:

- JSON parse/format
- YAML은 dependency 없이 lightweight indentation/key-value shape check
- formatted output 표시
- error state 표시

주의점:

- YAML full parser가 아니므로 anchors/custom tags/multi-doc YAML은 runtime parser 재검증을 권장하도록 설명했다.

블로그 글감:

- launch MVP에서 dependency를 늘리지 않고 useful enough한 validator를 만드는 기준
- "정확한 parser"와 "브라우저에서 빠르게 쓰는 실무 점검 도구" 사이의 trade-off

#### 8. Webhook Payload Formatter

파일:

- `src/components/tools/WebhookPayloadFormatter.tsx`

기능:

- webhook JSON payload parse/format
- event/id/timestamp/signature/amount/currency 같은 debugging field 추출
- payload size와 top-level keys 표시

블로그 글감:

- webhook debugging에서 먼저 봐야 할 필드
- formatter가 단순 pretty-print를 넘어서 diagnostic view가 되는 방법

#### 9. Cron Explainer

파일:

- `src/components/tools/CronExplainer.tsx`

기능:

- 5-field cron expression 해석
- preset expressions 제공
- timezone, day-of-month/day-of-week 조합, extended cron feature caveat 표시

빌드 중 처리한 이슈:

- JSX에서 `user's` apostrophe로 ESLint `react/no-unescaped-entities` 오류 발생
- `user&apos;s`로 수정

### Tier 2: Business calculators

#### 10. LLM Cost Calculator

파일:

- `src/components/tools/LlmCostCalculator.tsx`

기능:

- input tokens/request

- requests/day

블로그 글감:

#### 11. Automation ROI Calculator

파일:

- `src/components/tools/AutomationRoiCalculator.tsx`

기능:

- hours saved/run

- hourly value
- build cost
- monthly maintenance
- failure buffer

블로그 글감:

- 자동화는 "만들면 끝"이 아니라 maintenance/failure buffer까지 계산해야 한다

- nullable parsing helper 결과가 TypeScript narrowing 되지 않아 type check 실패

브랜드:

```text
Auto-Ops
Practical Tools. Agent Built.
```

기본 도메인:

```text
https://auto-ops.lotiony.com
```

홈의 방향:

- practical tools 중심 landing
- Tier 1 tool cards 노출
- build notes / posts는 보조 역할
- header nav도 Tools를 앞에 배치

이 변경은 중요하다. 같은 기능을 구현해도 첫 화면이 블로그처럼 보이면 사용자는 "읽을거리"로 인식한다. 첫 화면이 tools-first이면 사용자는 "쓸 수 있는 제품"으로 인식한다.

## 다음 작업 제안

우선순위는 다음 순서가 좋아 보인다.

### 1. Sitemap/robots/metadata 점검

도구 페이지가 실제로 검색엔진에 발견될 수 있어야 한다. tool registry 기반으로 sitemap에 모든 `/tools/[slug]`를 포함시키는 것이 먼저다.

### 2. Policy pages 구현

AdSense-ready 표면에는 Privacy/Terms/Contact/About/Disclaimer가 필요하다. 이건 화려한 기능은 아니지만 승인 가능성과 신뢰도를 높이는 기본 작업이다.

### 3. Guide layer 시작

각 도구가 thin content가 아니려면 tool page 자체 설명도 중요하지만, related guide가 실제로 존재해야 한다.

초기 guide 후보:

- LLM API 비용 계산 전 확인할 토큰 개념
- 자동화 ROI를 계산할 때 포함해야 하는 비용
- Webhook payload를 디버깅하는 기본 절차
- JSON과 YAML 설정 파일을 검증하는 실무 체크리스트
- Cron 표현식을 사람이 읽을 수 있게 해석하는 법
- UTM 링크를 만들 때 캠페인 이름을 정리하는 법
- Markdown 표를 깨지지 않게 만드는 법

### 4. Build log 글로 전환

이번 중간점검 자체가 블로그 글 초안이 될 수 있다.

가능한 제목:

- AI 에이전트로 Practical Tools Studio를 하루 만에 세우기
- 블로그를 도구 중심 사이트로 바꾼 과정: Auto-Ops 빌드 로그
- 17개 브라우저 도구를 만들며 배운 AI-assisted 개발의 현실
- AdSense-ready 도구 사이트를 만들 때 먼저 챙긴 것들

## 블로그 글로 쓸 때의 핵심 메시지

이 작업의 핵심은 "AI가 코드를 많이 썼다"가 아니다.

더 중요한 메시지는 다음이다.

1. 긴 계획보다 작은 검증 가능한 산출물이 중요하다.
2. AI 에이전트가 실패해도 fallback path가 있어야 한다.
3. 도구 사이트는 기능만이 아니라 설명, FAQ, 예시, metadata, 내부 링크까지 제품이다.
4. build/lint/type check는 AI 개발에서 더 중요해진다.
5. 한국형 UX는 단순 번역이 아니라 단위, 시간대, 환율, 생활 맥락을 반영하는 것이다.
6. 자동화/AI 도구를 만들 때는 비용과 ROI 계산기부터 필요하다.

## 개인 메모

이번 작업은 "AI로 무언가를 만든다"보다 "AI와 함께 계속 검증 가능한 상태를 유지하며 제품 표면을 만든다"에 가까웠다.

특히 인상적인 점:

- 17개 도구 자체보다, registry-driven 구조를 먼저 만든 것이 속도를 크게 올렸다.
- build가 실제로 여러 문제를 잡아줬다.
- subagent/Codex가 막혀도 작업이 멈추지 않도록 fallback을 둔 것이 중요했다.
- AdSense-ready는 광고 승인용 체크리스트가 아니라, 사용자가 신뢰할 만한 정보 구조를 만드는 일에 가깝다.

이 기록은 나중에 Auto-Ops의 첫 build note 또는 "AI 에이전트로 도구 사이트 만들기" 시리즈의 1편으로 확장할 수 있다.
